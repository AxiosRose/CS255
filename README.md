# CS 255 Portfolio Reflection

The DriverPass project involved analyzing business requirements and designing a web-based driver education management system for DriverPass. The goal was to help students prepare for DMV driving tests through online learning, practice exams, lesson scheduling, and behind-the-wheel instruction. Throughout the project, I used the client's interview and business requirements to ensure the design addressed both business goals and user expectations. The client wanted a system that would help students prepare for DMV driving tests through online course materials, practice exams, scheduling tools, and behind-the-wheel lesson management. The system also needed to support different user roles, including customers, secretaries, instructors, administrators, IT staff, and management.

One part I think I did particularly well was organizing the user needs into clear functional and nonfunctional requirements. I was able to connect the client’s goals to specific system features, such as account creation, package purchases, appointment scheduling, practice testing, progress tracking, report generation, and role-based access. I also think the UML diagrams helped show how the system would work from both the user and system side.

If I could revise one part of my work, I would improve the system design document by adding more detail to the UML diagrams and documenting additional relationships between system components. I would also include more design assumptions and edge cases so developers could better understand how the system should behave in uncommon situations.

I interpreted the user’s needs by reviewing what DriverPass wanted the system to accomplish and turning those needs into specific requirements and design features. For example, because customers needed to schedule lessons and take practice exams, I included scheduling workflows, practice exam activity diagrams, customer progress tracking, and database objects for appointments, lessons, exams, and payments. Considering user needs is important because software should solve the actual problem for the people using it. If the design does not reflect the needs of customers, staff, and administrators, the final system may be confusing, incomplete, or ineffective. At the same time, the design must balance those needs with technical requirements such as security, scalability, and maintainability to ensure the system remains effective over time.

My approach to designing software is to first understand the client’s problem, identify the main users, and define the stakeholder goals and business requirements before considering technical solutions. Once the requirements are documented, I use techniques such as use case diagrams, activity diagrams, sequence diagrams, and class diagrams to visualize system behavior and data relationships. This structured approach helps ensure that the final design remains aligned with user needs while also addressing security, scalability, and maintainability. In the future, I would continue using interviews, user stories, use case diagrams, activity diagrams, and class diagrams to analyze and design systems. Overall, this project reinforced the importance of careful planning before development begins. By creating detailed requirements and UML models first, I gained a better understanding of how thoughtful system design leads to software that is easier to build, maintain, and expand.
